## Что это

Внутренняя платформа ТОО «Caspian Integrated Services» (нефтесервис, Атырау).
Заменяет Documentolog и постепенно — старую ERP. Владелец: Жанибек Мубинов
(zh.mubinov@cis.kz). Разработчики: Диана Джумагазиева + джун + ИИ-ассистент.
Трёхъязычие: ru (основной) / en / kk.

## Стек и инфраструктура

- **Flask 3 + SQLAlchemy + PostgreSQL**, серверные Jinja-шаблоны (НЕ SPA —
  осознанное решение, не переписывать на React)
- **Хостинг:** Railway (сервис `web` + Postgres). Деплой: пуш в `main` →
  авто-деплой; старт-команда в настройках сервиса: `flask db upgrade &&
  gunicorn run:app ...` (миграции применяются сами; НЕ убирать db upgrade!)
- **Файлы:** Azure Blob Storage (`AZURE_STORAGE_CONNECTION_STRING`); без неё —
  локальный диск Railway, который стирается при деплое
- **Google Sheets** через сервисный аккаунт (`GOOGLE_CREDENTIALS_JSON`):
  3 таблицы — статусы техники («База»), реестр ТО (вкладка на машину),
  оплаты ПО. ID таблиц настраиваются в Админ-панель → Интеграции
- **ИИ:** Anthropic API (`ANTHROPIC_API_KEY`) — брифинг, ассистенты, агенты
- **Вход через Microsoft (Entra ID):** код готов, «спит» до задания
  `AZURE_CLIENT_ID/SECRET/TENANT_ID`
- **Фоновые задачи:** APScheduler внутри web-процесса, каждые 15 мин
  (синк таблиц, проверка ТО, оплаты); защита от двойного запуска —
  pg_advisory_lock; выключатель `ENABLE_SCHEDULER=0`
- **Бэкапы:** ночной GitHub Action `db-backup.yml` (pg_dump postgres:18,
  AES-256, артефакт 30 дней; секреты BACKUP_DATABASE_URL/BACKUP_PASSPHRASE)
  + бэкапы Railway. Восстановление: docs/RESTORE.md
- **CI:** GitHub Actions на каждый PR (pytest, 131 тест). Dependabot включён
- **Мониторинг:** Sentry-хук в app/__init__.py (активен при SENTRY_DSN)

## Ключевые доменные решения (почему так)

- **Маршрут согласования:** этапы согласующих (шаги 0..k-1, параллельно или
  последовательно) → подписывающий ВСЕГДА последним шагом. Форма передаёт
  stage_no[] + stage_reviewer_<n>[]. Движок в documents/helpers._build_route
- **Требование = doc_type `purchase_req`** (старый 'trebovanie' слит миграцией),
  нумерация ТМЦ-ГОД-№ через DocumentSequence (гонко-безопасно), сброс каждый год
- **Цикл ПО (услуги И товары):** Создание→Согласование→Подпись→На оплате
  (уведомление бухгалтерам-казначеям + их дашборд) →Оплачен (авто из таблицы
  оплат по номеру ПО, или вручную) →Закрывающие документы (дашборд автора)
  →Закрыт (отмечает бухгалтер)
- **Дефектные акты:** учётный код <код_машины>-ДА<n> (Б1-ДА1); закрывается
  записью ремонта в реестре ТО с упоминанием кода или кнопкой; из акта
  создаётся требование с префиллом запчастей
- **Синк с таблицами ОДНОСТОРОННИЙ** (таблицы → Postgres). Источник правды
  по статусам техники и ТО — таблицы + ТГ-бот. Обратная запись — в планах
- **Права:** роли × модули, редактируются в админке (role_permissions в БД,
  фолбэк на PERMISSIONS в коде); it_admin захардкожен на полный доступ;
  нельзя снять админа с себя / понизить последнего админа
- **Деньги:** только Numeric(14,2), НИКОГДА Float
- **Реквизиты ПО** упакованы строками «Ключ: значение» в Document.justification
  (историческое; распаковка _unpack_extras). Известный техдолг — при случае
  переехать на нормальные колонки/JSONB
- **Печать:** отдельные бланки print_po_services / print_po_goods /
  print_defect на макросах _print_po_base; НДС 16% от суммы при «Да»;
  лист согласования печатает фактическое состояние виз

## Правила командной работы

1. **Никогда не коммитить в main** — ветка → PR → зелёный CI → Merge
2. `git pull` на main перед созданием любой ветки
3. Один патч/фича = одна ветка = один PR; мержить до начала следующего
4. Ошибка «already exists in index» при git am = патч уже в main, не пересоздавать
5. Незавершённый am/rebase: `git am --abort`; грязная папка: `git reset --hard`
6. Тестовые роли смотреть через «Войти как» (админка → Пользователи), НЕ меняя
   свою роль
7. Dependabot: зелёный CI → merge; конфликт → комментарий `@dependabot recreate`

**Ждёт действий владельца:** SENTRY_DSN в Railway; ключи Entra ID от
IT-администратора; файлы знаний в «ИИ механика»; **доступ к OData 1С —
запросить у администратора 1С, это самая долгая зависимость**;
**таблица статей бюджета от финансистов** — получить файл/доступ, после
этого делаем синк статей и лимитов (см. бэклог, до тех пор статьи
ведутся вручную в админке); **таблица оплат ПО** — создать/расшарить и
прописать ID в Админка → Интеграции, без неё авто-отметка «Оплачен»
не работает (код синка готов, ждёт таблицу).

**Бэклог (приоритет сверху):** глобальный поиск Ctrl+K; KPI-дашборд директора;
контракты до конца (создание из UI, кварталы); HR (табель); email-уведомления;
API v1 для ТГ-ботов (боты → платформа вместо таблиц); 1С OData (read-only:
контрагенты, номенклатура, остатки, оплаты); RAG для агентов при росте базы
знаний; синк статей бюджета из таблицы финансистов (ОДНОСТОРОННИЙ, таблица →
Postgres, по образцу статусов техники; НЕ начинать до получения формата:
решить про коды статей vs имена, годовые vs месячные лимиты, и закрыть/

поиск с опечатками (pg_trgm); финальный редизайн через дизайн-скилл
(НЕ раньше стабилизации функционала); write-back в таблицы.

## Как работать над проектом с ИИ-ассистентом (Claude)

**Всегда Cowork, не обычный чат.** В Cowork ассистент клонирует репозиторий,
запускает тесты, собирает и проверяет патчи в песочнице — каждый ответ
проверен исполнением. В обычном чате он видит только текст, качество ответов
по коду принципиально ниже. Это главная причина разницы в эффективности.

**Стартовое сообщение новой сессии** (скопируй целиком):

> Прочитай PROJECT_NOTES.md в корне репозитория cis-platform и работай по
> правилам оттуда. Формат: изменения отдавай git-патчами (format-patch)
> по одному, один патч = одна ветка = один PR; перед выдачей прогоняй тесты;
> команды для меня пиши по одной строке под Windows cmd; отвечай по-русски;
> git-ошибки объясняй и сразу давай команды для починки.

**Контракт для ассистента:**
1. Перед сборкой патча — свежий fetch origin/main; ветка строго от него
2. Локально — затронутые тесты + смежные (полную сюиту гоняет CI на PR);
   полный локальный прогон — только при правках общего кода (models, helpers,
   config, миграции). Новый функционал = новые тесты. В тестовой конфигурации
   PBKDF2 = 1 итерация (PASSWORD_HASH_METHOD) — не «чинить» это на прод-настройку
3. Номер миграции — следующий после головной (см. migrations/versions)
4. Не объединять несколько фич в один патч
5. Ошибка «already exists in index» у пользователя = патч уже в main:
   собрать дельту, а не заставлять пересоздавать ветки
6. Крупные изменения — обновлять PROJECT_NOTES.md тем же патчем

**Экономия кредитов:** длинная сессия дорожает с каждым сообщением (весь
контекст прогоняется заново). Завершай сессию после 2–3 крупных фич и
начинай новую со стартового сообщения. Чужие репозитории смотри отдельной
короткой сессией с узким вопросом («как у них устроено X — дай выжимку и
3 идеи для нас»), не «посмотри репо».
